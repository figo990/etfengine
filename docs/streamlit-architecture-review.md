# Streamlit 架构升级评估

## 结论

当前不建议把 Dashboard 从 Streamlit 整体迁出，也不建议一次性重写前端。更合适的方向是继续做渐进式架构升级：

1. 保留 Streamlit 作为研究工作台和本地运维控制台。
2. 将耗时采集任务、数据健康判断、报告生成等能力继续下沉到服务层和任务层。
3. 页面层只保留输入、筛选、展示和任务提交，不直接承担长任务执行和持久化细节。

这个判断基于当前项目特点：页面以数据分析、表格、图表、手工运维为主，实时复杂交互和多人权限体系还不是核心矛盾。Streamlit 仍然能高效承载；真正需要升级的是应用分层和任务执行模型。

## 当前状态

- 一级导航已经收敛为 8 个工作流入口。
- 产业链研究、资讯事件、报告中心、数据管理已经形成统一工作台。
- 页面宽度已经切到宽屏布局，内容区最大宽度提升到 1760px。
- 页面级写库已收口到服务层。
- 数据健康检查、补采建议、更新日志和失败重试已经形成基础闭环。
- 后台任务注册器已升级为 DuckDB 持久化状态，重启后仍可查看任务历史。
- `估值与市场`、`产业链研究` 已从 legacy 嵌套改为原生工作台。

## 本轮架构升级

- `src/dashboard/task_runner.py`：提供后台任务提交、去重、状态查询和清理，任务状态写入 `dashboard_tasks`。
- 新闻监控与外盘季报接入统一刷新任务，并可通过 API、数据管理页、健康补采计划触发。
- 数据采集/补采类按钮统一提交后台任务，侧栏提供全局任务状态入口。
- 策略回测和报告生成已纳入后台任务，页面提供专门的结果回看与下载区域。
- 后台任务成功历史支持保留周期配置和调度清理；失败、中断记录保留用于排查。
- 数据管理 API 支持后台任务提交、列表查询和单任务轮询。
- 后台任务已增加 `task_type/tags`，页面和 API 不再只依赖 `task_key` 前缀筛选。
- 报告生成支持自动周报/月报调度，并在生成后接入配置化通知。
- 策略实验室支持参数模板和批量后台回测。
- `use_container_width` 已迁移为 `width="stretch"`，提前规避 Streamlit 兼容性告警。
- 新增页面冒烟测试和任务层测试，降低后续重构风险。

## 是否需要更大架构升级

### 暂不需要迁出 Streamlit

当前业务主要是：

- 本地 DuckDB 数据分析；
- 低频手工刷新；
- 定时采集与健康检查；
- 研究报告生成；
- 表格、图表和 Markdown 结果展示。

这些仍然是 Streamlit 的舒适区。迁移到 React/Vue + 独立后端会带来明显工程成本，但短期收益有限。

### 需要继续拆分 Streamlit 内部架构

建议把后续升级分为四层：

1. **页面层**：只负责布局、表单、筛选和结果展示。
2. **组件层**：统一指标卡、状态条、结果表、任务状态、空状态。
3. **服务层**：统一数据读取、配置写入、报告生成、健康判断。
4. **任务层**：统一长任务提交、执行、日志、失败重试和通知。

当前项目已经走到第 3 层和第 4 层的起点，后续应继续沿这个方向推进。

## 后续升级路线

1. 为后台任务增加更细的任务结果索引表，避免大结果长期存放在单行 JSON。
2. 为批量回测增加任务组 ID，支持一组任务的整体进度、汇总排序和一键保存方案。
3. 为自动报告增加页面化开关和时间编辑，写回配置文件。
4. 继续原生化剩余低频 legacy 页面，最终减少 `runpy` 嵌套加载。
5. 当出现多用户、权限、协作、复杂交互编辑器需求时，再评估独立前端。
